## Dépendances amont

### OpenBook (anciennement Serum)

- **Rôle** : carnet d'ordres limites centralisé. Les pools AMM v4 étaient initialement des pools hybrides utilisant OpenBook — les swaps pouvaient s'exécuter contre la courbe AMM du pool ou le marché OpenBook correspondant — mais **l'intégration OpenBook a depuis été désactivée**. Les pools AMM v4 ne publient plus ni ne maintiennent d'ordres sur OpenBook.
- **Où cela importe** : historiquement, seulement pour les pools AMM v4 ; aujourd'hui, aucun produit Raydium n'a de dépendance OpenBook active.
- **Surface de risque** : auparavant, les mises à jour ou pannes d'OpenBook pouvaient affecter le comportement d'AMM v4 (un tel incident en janvier 2023). Maintenant que l'intégration est désactivée, ce couplage est supprimé. CPMM et CLMM n'ont jamais eu de dépendance OpenBook.
- **Statut** : OpenBook continue de fonctionner sur Solana ; Raydium n'en dépend pas pour le trafic de swaps actuel. Les comptes de marché OpenBook référencés par les pools AMM v4 existants restent dans la chaîne comme état inerte pour la compatibilité rétroactive.

### Runtime Solana

- **Rôle** : le substrat d'exécution de tous les programmes Raydium.
- **Où cela importe** : partout. Le temps de finalité de Solana (~1s), les limites de calcul (1,4M CU par tx) et les tables de recherche d'adresses affectent tous directement l'expérience Raydium.
- **Statut** : Solana se met à jour périodiquement ; Raydium suit les versions des clients validateurs (Agave, Firedancer) et ajuste si le comportement change.

### Programmes de tokens (SPL Token et Token-2022)

- **Rôle** : les primitives de tokens que tous les pools Raydium détiennent.
- **SPL Token** : le programme hérité ; fonctionne sur tous les produits.
- **Token-2022** : programme avec extensions ; fonctionne sur CPMM, CLMM, Farm v6, pas AMM v4.
- **Surface de risque** : le comportement des extensions (frais de transfert, hooks, etc.) peut affecter la fonctionnalité du pool. Voir [`security/oracle-and-token-risks`](/fr/security/oracle-and-token-risks).

## Intégrations aval

### Jupiter et autres agrégateurs

- **Rôle** : router les trades sur plusieurs DEX Solana, y compris Raydium.
- **Mode d'intégration** : Jupiter lit l'état du pool depuis l'API REST de Raydium et via des appels directs on-chain, puis compose les instructions de swap via le SDK Raydium (ou parfois directement).
- **Partage du volume** : environ 60% du volume de Jupiter passe par les pools Raydium au moins une fois.
- **Chemin de l'intégrateur** : le modèle général d'agrégateur est documenté dans [`integration-guides/aggregator`](/fr/integration-guides/aggregator).

### Portefeuilles

- **Rôle** : permettre aux utilisateurs de swapper, suivre les positions LP et participer aux fermes sans quitter le portefeuille.
- **Mode d'intégration** :
  - Swap : router via Jupiter ou directement via le SDK Raydium.
  - Affichage des positions : énumérer les ATA de l'utilisateur et les participations NFT, puis les recouper avec la liste des pools de Raydium ([`integration-guides/wallet-integration`](/fr/integration-guides/wallet-integration)).
  - Affichage des fermes : appeler le point de terminaison `/positions/staking` de Raydium.
- **Impact utilisateur** : tout portefeuille disposant d'un swap in-app route probablement via Raydium sans que vous le remarquiez.

### Protocoles de prêt (Kamino, Marginfi, Drift)

- **Rôle** : protocoles d'emprunt/prêt qui ont besoin de prix oracle et de lieux d'exécution pour les liquidations.
- **Mode d'intégration** : utiliser les TWAP d'ObservationState de CLMM comme oracles ; swaps directs via Raydium pour la conversion des collatéraux de liquidation.
- **Risque** : si un protocole de prêt utilise un prix spot Raydium au lieu d'un TWAP, il peut être exploité par des manipulations de type flash-loan (voir [`security/attack-vectors`](/fr/security/attack-vectors)).

### Chercheurs MEV et arbitrageurs

- **Rôle** : maintenir les prix alignés sur Raydium, Orca, Phoenix et les bourses centralisées.
- **Effet sur les utilisateurs** : prix serrés la plupart du temps ; attaques sandwich sur les swaps moins protégés. Voir [`integration-guides/routing-and-mev`](/fr/integration-guides/routing-and-mev).

### Bots (market makers, exécuteurs DCA, liquidateurs)

- **Rôle** : fournir de la liquidité (LP sur des pools peu profonds), exécuter les trades programmés, fermer les positions.
- **Chemin d'intégration** : généralement des bots Python ou Rust utilisant [`sdk-api/python-integration`](/fr/sdk-api/python-integration) ou des wrappers CPI Rust.

## Position dans le volume agrégé

En avril 2026 :

- Raydium traite ~35–45% du volume DEX à l'échelle de Solana sur n'importe quel jour donné.
- Part de marché maximale ~55% pendant les périodes où l'activité memecoin se concentre dans LaunchLab + pools diplômés.
- Minimum bas ~25% pendant les périodes d'activité importante basée sur le carnet (les grands flux institutionnels se dirigent vers Phoenix lorsqu'il est disponible).

Ces chiffres fluctuent ; le point est que Raydium est de manière fiable l'un des deux meilleurs lieux par volume.

## Écosystème Token-2022

Raydium est l'un des premiers grands DEX à supporter nativement Token-2022. Implications :

- Les nouvelles émissions avec frais de transfert (par exemple, les royalties, les tokens à frais pour l'émetteur) peuvent être LP'd.
- Les émissions avec transfer-hook peuvent exister (bien qu'avec des avertissements de risque).
- Les émissions non-transférables et les délégations permanentes sont bloquées.

À mesure que davantage de projets se lancent sous Token-2022 — en particulier la foule « institutionnelle » utilisant des émissions réglementées — le support de Raydium le positionne comme un lieu pour cette classe de liquidité.

## Dépendances orientées vers l'avenir

Points à surveiller qui pourraient changer la position de Raydium dans l'écosystème :

- **Diversification des validateurs Solana** : si Firedancer atteint la domination, les caractéristiques de performance pourraient changer.
- **Pontage cross-chain** : Raydium a étudié les extensions vers d'autres L1/L2. Aucun déploiement pour l'instant.
- **Évolution de Jito** : l'intégration continue entre les moteurs de bloc et la DeFi pourrait changer la dynamique du MEV.
- **Carnets d'ordres on-chain (Phoenix)** : la maturation continue pourrait déplacer une partie du flux semblable à AMM vers les carnets.
