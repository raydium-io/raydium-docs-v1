## 本文件的目的

本頁是 Raydium 文檔集組織方式的**單一來源**。在為任何章節撰寫詳細內容之前，請參考本文件——它定義了讀者分層、章節邊界、必須保持一致的交叉參考，以及磁碟上的目錄配置。

這是刻意設計為內部結構文件。面向讀者的登陸頁面位於 `index.mdx`。

## 讀者模型

文檔同時服務於三個讀者群體。下面每個章節都標有其主要服務的讀者群體，這樣撰寫者就知道應該針對的語氣和深度。

<CardGroup cols={3}>
  <Card title="開發者／整合商" icon="code">
    智能合約開發者、DApp／聚合器整合商、SDK 使用者。需要鏈上帳戶配置、指令、REST／交易 API、SDK 和可執行代碼。
  </Card>
  <Card title="協議研究人員／審計者" icon="flask">
    研究人員、安全審計員、DeFi 分析師。需要算法、不變量、帳戶／狀態機、手續費經濟、RAY 代幣經濟及風險面。
  </Card>
  <Card title="終端使用者" icon="user">
    流動性提供者、交易者（現貨和永續）和代幣創建者。需要面向任務的演練：交換、增加／移除流動性、建立資金池、領取獎勵、交易永續、透過 LaunchLab 啟動代幣。
  </Card>
</CardGroup>

一個分層的讀者不應該在另一個分層的頁面上感到迷茫。跨讀者群體導航通過以下方式解決：(a) 章節範圍界定，以及 (b) 每個章節登陸頁面上清晰的「如果你想要 X，請閱讀第 Y 章」標註。

## 設計原則

1. **一個主題，一個主頁。** 一個概念在恰好一個章節中定義，並從其他章節連結。例如，手續費在每個產品中*定義*（每個產品都有自己的手續費模型），但*比較手續費表*位於 `reference/fee-comparison`。
2. **漸進式深度。** 每個產品章節以一段簡潔英文概述開始，然後進入帳戶 → 數學 → 指令 → 手續費 → 代碼。讀者可以在任何深度停止。
3. **重要之處都有可執行代碼。** 每個指令級部分都以最少的、可複製貼上的 TS／Rust／Python 範例結尾。長篇演示位於每個產品的 `code-demos.mdx` 下，這樣敘述就不會被代碼淹沒。
4. **版本固定。** SDK 版本、程序 ID 和 Solana 群集在任何顯示代碼的頁面頂部命名。過時的範例是 DeFi 文檔的第一大失敗模式。
5. **將「如何運作」與「如何使用」分開。** 算法有自己的章節；使用流程有自己的章節。產品章節雙向連結。
6. **安全是第一類章節，而不是腳註。** 審計員不應該必須在產品章節中搜尋。
7. **讀者分層的使用流程。** 終端使用者演練按讀者角色分組——交易者、流動性提供者、代幣創建者——而不是按程序或功能。

## 每個產品的模板

為了保持產品線的可比性，每個產品檔案夾都遵循相同的結構。不暴露鏈上帳戶的程序（例如永續，它是 Orderly Network 上的白標後端）使用等效的表面特定檔案。

| 檔案 | 目的 | 必需條件 |
|-----|------|---------|
| `index.mdx` | 章節登陸：一段落介紹、何時使用、子頁面連結、關鍵程序 ID。 | 全部 |
| `overview.mdx` | 概念模型——此產品做什麼以及為什麼，伴有圖表。 | 鏈上產品 |
| `accounts.mdx` | 每個鏈上帳戶：PDA 種子、欄位配置、不變量、IDL 參考。 | 鏈上產品 |
| `math.mdx` | 不變量、定價公式、數值範例。交叉連結到 `algorithms/`。 | 鏈上產品 |
| `instructions.mdx` | 完整指令參考：引數、帳戶列表、前置／後置條件、錯誤代碼。 | 鏈上產品 |
| `fees.mdx` | 手續費等級、手續費分配（LP／協議／基金／創建者）；對於 CLMM，還有獎勵發放，因為 CLMM 的獎勵系統重用每個頭寸費用成長會計。 | 全部 |
| `code-demos.mdx` | 可複製貼上的 TypeScript + Rust 範例，用於產品暴露的每個流程。 | 全部 |

每個產品的擴展，其中標準模板不夠：

- **CLMM** 增加了 `ticks-and-positions.mdx`，因為 tick 數學太大無法嵌入 `math.mdx`。
- **LaunchLab** 增加了 `bonding-curve.mdx`、`creator-fees.mdx`、`global-config.mdx`、`platform-config.mdx`、`platforms.mdx`、`vesting.mdx`，加上 `tips-and-gotchas/` 子檔案夾。
- **Perps** 使用 `trading-basics.mdx`、`order-types.mdx`、`collateral.mdx`、`fees.mdx`、`code-demos.mdx`——沒有 `accounts.mdx`／`math.mdx`／`instructions.mdx`，因為會場是鏈下（Orderly Network）的，附帶 Solana 邊包裝。

## 跨章節約定

- **程序 ID 和群集**始終在 `reference/program-addresses.mdx` 中定義。其他頁面連結到那裡，而不是硬編碼值。
- **錯誤代碼**始終在 `reference/error-codes.mdx` 中定義。指令頁面連結到特定的錨點。
- **數學**位於 `algorithms/`。產品 `math.mdx` 頁面給出產品特定的*實例化*並連結回。
- **手續費經濟**按意圖分成兩個主頁：每個產品的機制在 `products/<x>/fees.mdx` 中，協議收入／RAY 回購流在 `ray/` 中。跨產品比較表位於 `reference/fee-comparison`。
- **RAY 代幣經濟**位於 `ray/`。其他地方提及 RAY（質押流程、手續費分配、治理）的頁面連結到那裡。
- **代碼版本化。** 每個代碼塊都有一個版本橫幅：SDK 版本、程序 ID、Solana 群集、上次驗證日期。
- **圖表。** 架構圖是簽入 `/images/architecture/` 的 SVG。源代碼（Mermaid 或 draw.io）在 SVG 旁邊提交。
- **使用流程按讀者標記，而不是按功能標記。** 演練屬於「交易者用／流動性提供者用／代幣創建者用」——*而不是*它觸及的程序。

## 多語言

預設語言（英文）位於倉庫根目錄，沒有路徑前綴。每個非英文語言位於其自己的頂級檔案夾（`zh/`、`zh-Hant/`、`ja/`、`ko/`、`ru/`、`es/`、`de/`、`fr/`）並鏡像英文樹。

- **內容（MDX）：** 由 Mintlify 自動翻譯保持同步。新的／更改的英文頁面會自動翻譯並推送到每個語言檔案夾。
- **導航（`docs.json`）：** *不*自動翻譯。每個語言在 `navigation.languages` 下有自己的 `tabs` 塊，群組標籤、分頁名稱和頁面列表必須在英文導航結構改變時手動更新。
- **分頁和群組標籤**使用語言自己的語言；**頁面路徑**用語言代碼前綴（例如 `zh/products/cpmm/overview`）。
- **舊版 URL 處理。** `/en/<path>` 重定向到 `/<path>`（在 `docs.json` `redirects` 中配置）。當頁面移動或重命名時，添加重定向項而不是依賴自動翻譯器。

## 明確不在範圍內

如果使用者在文檔中詢問以下內容，應該拒絕，因為這些明確不在本文檔集中涵蓋：

- 交易策略／市場建議。
- 代幣價格預測。
- 詳細的代幣經濟建模，超過閱讀代碼所需的內容。
- 包含機密或管理程序的內部運行手冊——這些位於私有倉庫中。

## 狀態

這是架構的 v0.2。與 v0.1 相比：

- 舊的 `operations/` 章節（DAO、財務、費用參數、緊急程序）已被 `ray/` 取代，範圍圍繞 RAY 代幣經濟和協議收入，並移至**安全與運營**分頁。
- 三條新的產品線——`perps/`、`stable/`、`routing/`——是 `products/` 下的一級條目。
- `getting-started/`、`quick-start/`、`resources/` 和 `api-reference/` 現在是一級頂級章節。
- `user-flows/` 在導航中按讀者角色（交易者／流動性提供者／代幣創建者）而不是按功能重新組織。
- LaunchLab 現在有專門的**啟動工具**分頁和 `tips-and-gotchas/` 子檔案夾，用於操作邊界情況。
- 多語言支援已正式確定：英文是倉庫根目錄的預設，有八個語言鏡像由 Mintlify 自動翻譯內容和手動導航進行維護。
