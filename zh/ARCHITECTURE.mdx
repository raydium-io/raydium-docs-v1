## 本文档的用途

本页是 Raydium 文档集组织方式的**唯一真实来源**。在为任何章节编写详细内容之前，请查阅本文档——它定义了受众轨迹、章节边界、必须保持一致的交叉引用以及磁盘上的目录布局。

这是一份有意为之的内部结构文档。面向读者的登陆页面位于 `index.mdx`。

## 受众模型

文档同时服务于三类受众。下面每个章节都标有该章节主要服务的受众，以便作者了解应该采用的语调和深度。

<CardGroup cols={3}>
  <Card title="开发者 / 集成商" icon="code">
    智能合约开发者、DApp/聚合器集成商、SDK 用户。需要链上账户布局、指令、REST/交易 API、SDK 和可运行的代码。
  </Card>
  <Card title="协议研究员 / 审计师" icon="flask">
    研究员、安全审计师、DeFi 分析师。需要算法、不变量、账户/状态机、费用经济学、RAY 代币经济学和风险表面。
  </Card>
  <Card title="终端用户" icon="user">
    流动性提供者、交易者（现货和永续）和代币创建者。需要面向任务的演练：交换、添加/移除流动性、创建池、领取奖励、交易永续、通过 LaunchLab 启动代币。
  </Card>
</CardGroup>

一个轨迹的读者在阅读另一个轨迹的页面时应该感到自在。跨受众导航通过以下方式解决：(a) 章节范围划分和 (b) 每个章节登陆页面上的显著「如果你想要 X，请阅读第 Y 章」标注。

## 设计原则

1. **一个主题，一个家。** 一个概念恰好在一个章节中定义，并从其他章节链接到它。例如，费用在每个产品中定义（每个产品有自己的费用模型），但**对比费用表**位于 `reference/fee-comparison`。
2. **渐进式深度。** 每个产品章节以一段纯英文概述开始，然后进入账户 → 数学 → 指令 → 费用 → 代码。读者可以在任何深度停止。
3. **到处都有可运行的代码。** 每个指令级部分都以最小的、可复制粘贴的 TS / Rust / Python 示例结尾。长演示位于每个产品的 `code-demos.mdx` 下，这样叙述就不会被代码淹没。
4. **版本固定。** 显示代码的任何页面顶部都命名了 SDK 版本、程序 ID 和 Solana 集群。陈旧的示例是 DeFi 文档的首要失败模式。
5. **区分"它如何工作"和"如何使用它"。** 算法位于自己的章节中；用户流位于自己的章节中。产品章节两种方式都链接。
6. **安全是一流章节，而不是脚注。** 审计师不应该需要在产品章节中 grep。
7. **按受众分段的用户流。** 终端用户演练按读者角色分组——交易者、流动性提供者、代币创建者——而不是按程序或功能分组。

## 每个产品的模板

为了保持产品线的可比性，每个产品文件夹遵循相同的结构。不公开链上账户的程序（例如 Perps，它是 Orderly Network 上的白标后端）用等效的表面特定文件替代。

| 文件 | 用途 | 必需对象 |
|-----|-----|--------|
| `index.mdx` | 章节登陆页：一段概述、何时使用、子页面链接、关键程序 ID。 | 全部 |
| `overview.mdx` | 概念模型——这个产品做什么和为什么，附带图表。 | 链上产品 |
| `accounts.mdx` | 每个链上账户：PDA 种子、字段布局、不变量、IDL 参考。 | 链上产品 |
| `math.mdx` | 不变量、定价公式、数值示例。交叉链接到 `algorithms/`。 | 链上产品 |
| `instructions.mdx` | 完整指令参考：参数、账户列表、前置/后置条件、错误代码。 | 链上产品 |
| `fees.mdx` | 费用等级、费用分割（LP / 协议 / 基金 / 创建者）；对于 CLMM，也有奖励排放，因为 CLMM 的奖励系统重用每个头寸的费用增长计费。 | 全部 |
| `code-demos.mdx` | 可复制粘贴的 TypeScript + Rust 示例，用于产品公开的每个流。 | 全部 |

每个产品的扩展，当标准模板不够时：

- **CLMM** 添加 `ticks-and-positions.mdx`，因为刻度数学太大，无法嵌入 `math.mdx`。
- **LaunchLab** 添加 `bonding-curve.mdx`、`creator-fees.mdx`、`global-config.mdx`、`platform-config.mdx`、`platforms.mdx`、`vesting.mdx`，加上 `tips-and-gotchas/` 子文件夹。
- **Perps** 使用 `trading-basics.mdx`、`order-types.mdx`、`collateral.mdx`、`fees.mdx`、`code-demos.mdx`——没有 `accounts.mdx` / `math.mdx` / `instructions.mdx`，因为场地是链外（Orderly Network）的，带有 Solana 端包装。

## 跨章节约定

- **程序 ID 和集群**始终在 `reference/program-addresses.mdx` 中定义。其他页面链接到那里，而不是硬编码值。
- **错误代码**始终在 `reference/error-codes.mdx` 中定义。指令页面链接到特定的锚点。
- **数学**位于 `algorithms/`。产品 `math.mdx` 页面给出产品特定的**实例化**并链接回。
- **费用经济学**按意图分为两个家：每个产品在 `products/<x>/fees.mdx` 中的机制，以及协议收入 / RAY 回购流在 `ray/`。跨产品对比表位于 `reference/fee-comparison`。
- **RAY 代币经济学**位于 `ray/`。其他提及 RAY 的页面（质押流、费用分割、治理）链接到那里。
- **代码版本化。** 每个代码块都有版本横幅：SDK 版本、程序 ID、Solana 集群、最后验证日期。
- **图表。** 架构图是 SVG，检入 `/images/architecture/`。源（Mermaid 或 draw.io）在 SVG 旁边提交。
- **用户流按受众标记，而不是按功能标记。** 演练属于「对交易者 / 对流动性提供者 / 对代币创建者」——**不是**它所涉及的程序下。

## 多语言

默认语言（英文）位于仓库根目录，无路径前缀。每个非英文语言位于其自己的顶级文件夹（`zh/`、`zh-Hant/`、`ja/`、`ko/`、`ru/`、`es/`、`de/`、`fr/`）并镜像英文树。

- **内容（MDX）：** 由 Mintlify 的自动翻译保持同步。新的/更改的英文页面自动翻译并推送到每个语言文件夹。
- **导航（`docs.json`）：** **不**自动翻译。每种语言在 `navigation.languages` 下有自己的 `tabs` 块，当英文导航结构变化时，组标签、标签名和页面列表必须手动更新。
- **标签和组标签**使用该语言的自身语言；**页面路径**以语言代码为前缀（例如，`zh/products/cpmm/overview`）。
- **旧版 URL 处理。** `/en/<path>` 重定向到 `/<path>`（在 `docs.json` `redirects` 中配置）。当页面移动或重命名时，添加重定向条目而不是依赖自动翻译器。

## 明确超出范围的内容

以下内容**不**在本文档集中涵盖，如果用户在文档中要求应该拒绝：

- 交易策略 / 市场建议。
- 代币价格预测。
- 详细的代币经济学建模，超过读代码所需。
- 包含秘密或管理程序的内部运行手册——这些位于私有仓库。

## 状态

这是架构的 v0.2。与 v0.1 相比：
