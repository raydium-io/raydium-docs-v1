## 이 문서의 목적

이 페이지는 Raydium 문서 세트가 어떻게 조직되어 있는지에 대한 **권위 있는 정보 출처**입니다. 어떤 챕터에 대한 자세한 콘텐츠를 작성하기 전에 이 문서를 참고하세요. 이 문서에는 사용자 추적(audience track), 챕터 경계, 일관성을 유지해야 하는 상호 참조, 디스크의 디렉토리 레이아웃이 정의되어 있습니다.

의도적으로 내부적이고 구조적인 문서입니다. 독자 대면 랜딩 페이지는 `index.mdx`에 있습니다.

## 사용자 모델

문서는 세 가지 사용자 그룹을 병렬로 지원합니다. 아래의 모든 챕터는 주로 어떤 사용자 그룹을 위한 것인지 표시되어 있으므로, 작성자는 어떤 톤과 깊이로 작성할지 알 수 있습니다.

<CardGroup cols={3}>
  <Card title="개발자 / 통합자" icon="code">

  </Card>
  <Card title="프로토콜 연구자 / 감사자" icon="flask">

  </Card>

한 추적의 독자가 다른 추적의 페이지에서 길을 잃어서는 안 됩니다. 사용자 간 네비게이션은 (a) 챕터 범위 지정과 (b) 모든 챕터 랜딩 페이지의 눈에 띄는 "X를 원한다면 챕터 Y를 읽으세요"라는 설명으로 해결됩니다.

## 설계 원칙

1. **한 주제, 한 곳의 정의.** 개념은 정확히 하나의 챕터에서 정의되고 다른 곳에서 링크됩니다. 예를 들어, 수수료는 제품별로 *정의*되지만(각 제품은 고유한 수수료 모델을 가짐), *비교 수수료 표*는 `reference/fee-comparison`에 위치합니다.
2. **점진적 깊이.** 각 제품 챕터는 한 문단의 평문 개요로 시작한 후 계정 → 수학 → 명령어 → 수수료 → 코드로 진행됩니다. 독자는 어느 깊이에서든 멈출 수 있습니다.
3. **중요한 곳마다 실행 가능한 코드.** 모든 명령어 수준의 섹션은 최소한의, 복사해서 붙여넣을 수 있는 TS / Rust / Python 예제로 끝납니다. 긴 데모는 각 제품의 `code-demos.mdx` 아래에 있으므로 서술이 코드에 묻히지 않습니다.
4. **버전 고정.** SDK 버전, 프로그램 ID, Solana 클러스터는 코드를 표시하는 모든 페이지의 상단에 이름이 지정됩니다. 오래된 예제는 DeFi 문서의 #1 실패 원인입니다.
5. **"어떻게 작동하는가"와 "어떻게 사용하는가"를 분리합니다.** 알고리즘은 자신의 챕터에서 살고, 사용자 흐름은 자신의 챕터에서 삽니다. 제품 챕터는 양방향으로 링크합니다.
6. **보안은 1급 챕터이지, 각주가 아닙니다.** 감사자는 제품 챕터를 검색하지 않아도 됩니다.
7. **사용자 역할별 분류된 사용자 흐름.** 최종 사용자 안내는 프로그램이나 기능별이 아니라 독자 역할(거래자, 유동성 공급자, 토큰 생성자)별로 그룹화됩니다.

## 최상위 네비게이션

사이트는 8개의 최상위 탭으로 구성되어 있습니다(`docs.json` 참고). 각 탭은 디스크의 챕터 또는 챕터 클러스터에 매핑됩니다:

| 탭 | 사용자 | 포함 내용 |
|-----|----------|------------------|
| **Overview** | 모두 | `introduction/`, `protocol-overview/` — Raydium이 무엇인지, 프로그램 그래프가 어떻게 맞는지. |
| **User Guide** | 최종 사용자 | `getting-started/`, `user-flows/` — 온보딩 및 독자 역할별 작업 지향 안내(거래자 / 유동성 공급자 / 토큰 생성자). |
| **Products** | 개발자, 연구자 | `products/`, `algorithms/` — 프로그램당 하나의 하위 챕터, 그리고 심화 수학 챕터. |
| **Launch tools** | 생성자, 개발자 | `products/launchlab/`, `user-flows/launch-token-launchlab`, `products/launchlab/tips-and-gotchas/` — LaunchLab 프로그램 및 운영 문제. |
| **Developer Guide** | 개발자 | `solana-fundamentals/`, `quick-start/`, `sdk-api/`, `integration-guides/` — 기본 사항, 실습 빠른 시작, SDK/API 참조, 통합 패턴. |
| **Security & Ops** | 연구자, 운영자 | `security/`, `ray/` — 감사, 위험 표면, 공격 벡터, RAY 토큰노믹스, 재무, 환매, 스테이킹, 프로토콜 수익 경제학. |
| **API Reference** | 개발자 | `api-reference/` — 서비스별 개요 페이지(REST v1/v3, Trade, Perps, LaunchLab 서비스 API, Owner, Dynamic IPFS, Temp). |
| **Reference** | 모두 | `reference/`, `resources/` — 프로그램 주소, 오류 코드, 용어집, 수수료 비교, 변경 로그, 브랜드 키트, 커뮤니티 도구. |

```
                                      # Mintlify 자동 번역은 MDX 콘텐츠를 동기화 상태로 유지합니다.
                                      # docs.json의 네비게이션은 각 로캘에 대해
                                      # 수동으로 업데이트되어야 합니다.
```

## 제품별 템플릿

제품 라인들을 비교 가능하게 유지하기 위해 모든 제품 폴더는 같은 형태를 따릅니다. 온체인 계정을 노출하지 않는 프로그램(예: Perps, Orderly Network 위의 화이트라벨 백엔드)은 동등한 서비스별 파일로 대체합니다.

| 파일 | 목적 | 필수 |
|------|---------|--------------|
| `index.mdx` | 챕터 랜딩: 한 문단 소개, 사용 시기, 하위 페이지 링크, 주요 프로그램 ID. | 모두 |
| `overview.mdx` | 개념 모델 — 이 제품이 무엇을 하고 왜 하는지, 다이어그램 동반. | 온체인 제품 |
| `accounts.mdx` | 모든 온체인 계정: PDA 시드, 필드 레이아웃, 불변성, IDL 참조. | 온체인 제품 |
| `math.mdx` | 불변성, 가격 공식, 수치 예제. `algorithms/`로 상호 링크. | 온체인 제품 |
| `instructions.mdx` | 완전한 명령어 참조: 인자, 계정 목록, 전/후 조건, 오류 코드. | 온체인 제품 |
| `fees.mdx` | 수수료 티어, 수수료 분할(LP / 프로토콜 / 펀드 / 생성자); CLMM의 경우, 보상 배출, CLMM의 보상 시스템이 포지션별 수수료 성장 회계를 재사용하므로. | 모두 |
| `code-demos.mdx` | 제품에서 노출하는 모든 흐름에 대해 복사해서 붙여넣을 수 있는 TypeScript + Rust 예제. | 모두 |

표준 템플릿으로 충분하지 않은 제품별 확장:

- **CLMM**은 `ticks-and-positions.mdx`를 추가합니다. 틱 수학이 `math.mdx`에 포함하기에 너무 크기 때문입니다.
- **LaunchLab**은 `bonding-curve.mdx`, `creator-fees.mdx`, `global-config.mdx`, `platform-config.mdx`, `platforms.mdx`, `vesting.mdx`, 그리고 `tips-and-gotchas/` 서브폴더를 추가합니다.
- **Perps**는 `trading-basics.mdx`, `order-types.mdx`, `collateral.mdx`, `fees.mdx`, `code-demos.mdx`를 사용합니다 — 거래장이 온체인이 아닌 오프체인(Orderly Network)이므로 `accounts.mdx` / `math.mdx` / `instructions.mdx`가 없습니다.

## 챕터 간 규칙

- **프로그램 ID 및 클러스터**는 항상 `reference/program-addresses.mdx`에서 정의됩니다. 다른 페이지는 여기에 링크하고 값을 하드코드하지 않습니다.
- **오류 코드**는 항상 `reference/error-codes.mdx`에서 정의됩니다. 명령어 페이지는 특정 앵커로 링크합니다.
- **수학**은 `algorithms/`에 있습니다. 제품 `math.mdx` 페이지는 제품별 *인스턴스화*를 제공하고 다시 링크합니다.
- **수수료 경제학**은 의도별로 두 곳에 분산됩니다: 제품별 메커니즘은 `products/<x>/fees.mdx`에, 프로토콜 수익 / RAY 환매 흐름은 `ray/`에 있습니다. 교차 제품 비교 표는 `reference/fee-comparison`에 있습니다.
- **RAY 토큰노믹스**는 `ray/`에 있습니다. RAY를 언급하는 다른 곳(스테이킹 흐름, 수수료 분할, 거버넌스)은 여기에 링크합니다.
- **코드 버전 지정.** 모든 코드 블록에는 버전 배너가 있습니다: SDK 버전, 프로그램 ID, Solana 클러스터, 마지막 검증 날짜.
- **다이어그램.** 아키텍처 다이어그램은 `/images/architecture/`에 체크인된 SVG입니다. 소스(Mermaid 또는 draw.io)는 SVG 옆에 커밋됩니다.
- **사용자 흐름은 기능이 아니라 사용자별로 태그됩니다.** 안내는 "거래자용 / 유동성 공급자용 / 토큰 생성자용" 아래에 속합니다 — *not* 건드리는 프로그램 아래.

## 다국어

기본 로캘(영어)은 경로 접두사 없이 저장소 루트에 있습니다. 각 영어가 아닌 로캘은 자신의 최상위 폴더(`zh/`, `zh-Hant/`, `ja/`, `ko/`, `ru/`, `es/`, `de/`, `fr/`) 아래에 있고 영어 트리를 미러합니다.

- **콘텐츠(MDX):** Mintlify의 자동 번역으로 동기화 상태로 유지됩니다. 새로운/변경된 영어 페이지는 자동으로 각 로캘 폴더로 번역되어 푸시됩니다.
- **네비게이션(`docs.json`):** 자동 번역되지 않습니다. 각 언어는 `navigation.languages` 아래에 자신의 `tabs` 블록을 가지며, 그룹 레이블, 탭 이름, 페이지 목록은 영어 네비게이션이 구조적으로 변할 때마다 수동으로 업데이트되어야 합니다.
- **탭 및 그룹 레이블**은 로캘 고유의 언어입니다. **페이지 경로**는 로캘 코드로 접두사됩니다(예: `zh/products/cpmm/overview`).
- **레거시 URL 처리.** `/en/<path>`는 `/<path>`로 리디렉트됩니다(`docs.json` `redirects`에 구성됨). 페이지가 이동하거나 이름이 바뀔 때 자동 번역기에 의존하지 말고 리디렉트 항목을 추가하세요.

## 명시적으로 범위를 벗어난 것

다음은 이 문서 세트에서 *다루지 않으며* 사용자가 인문서 내에서 요청하면 거절해야 합니다:

- 거래 전략 / 시장 조언.
- 토큰 가격 예측.
- 코드를 읽는 데 필요한 것 이상의 상세한 토큰노믹스 모델링.
- 비밀이나 관리 절차를 포함하는 내부 실행 설명서 — 그것들은 개인 저장소에 있습니다.

## 상태

이것은 아키텍처 v0.2입니다. v0.1과 비교하면:

- 기존 `operations/` 챕터(DAO, 재무, 수수료 매개변수, 긴급 절차)가 `ray/`로 대체되었으며, RAY 토큰노믹스 및 프로토콜 수익을 중심으로 범위가 지정되고, **Security & Ops** 탭으로 이동했습니다.
- 세 개의 새로운 제품 라인 — `perps/`, `stable/`, `routing/` — 은 `products/` 아래의 1급 항목입니다.
- `getting-started/`, `quick-start/`, `resources/`, `api-reference/`는 이제 1급 최상위 챕터입니다.
- `user-flows/`는 기능이 아니라 사용자 역할(거래자 / 유동성 공급자 / 토큰 생성자)별로 네비게이션에서 재구성되었습니다.
- LaunchLab은 이제 전용 **Launch tools** 탭과 운영 가장자리 케이스를 위한 `tips-and-gotchas/` 서브폴더를 가집니다.
- 다국어 지원이 공식화되었습니다: 영어는 저장소 루트의 기본값이며, Mintlify 자동 번역으로 콘텐츠를 유지하고 네비게이션을 수동으로 유지하는 8개의 로캘 미러가 있습니다.

리프 페이지 작성은 챕터별로 계속됩니다. 스캐폴딩으로 존재하는 스텁은 프론트매터의 `status: draft` 노트로 표시됩니다.
