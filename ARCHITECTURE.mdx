## Purpose of this document

This page is the **source of truth** for how the Raydium documentation set is organized. Before writing detailed content for any chapter, consult this document вЂ” it defines the audience tracks, the chapter boundaries, the cross-references that must stay consistent, and the directory layout on disk.

It is deliberately an internal, structural document. The reader-facing landing page lives in `index.mdx`.

## Audience model

The docs serve three audiences in parallel. Every chapter is tagged below with which audience(s) it primarily serves, so writers know the voice and depth to aim for.

<CardGroup cols={3}>
  <Card title="Developers / Integrators" icon="code">
    Smart-contract developers, DApp/aggregator integrators, SDK users. Need on-chain account layouts, instructions, REST/Trade APIs, SDKs, and runnable code.
  </Card>
  <Card title="Protocol researchers / Auditors" icon="flask">
    Researchers, security auditors, DeFi analysts. Need algorithms, invariants, account/state machines, fee economics, RAY tokenomics, and risk surface.
  </Card>
  <Card title="End users" icon="user">
    Liquidity providers, traders (spot and perps), and token creators. Need task-oriented walkthroughs: swap, add/remove liquidity, create a pool, claim rewards, trade perps, launch a token via LaunchLab.
  </Card>
</CardGroup>

A reader of one track should never feel lost in another track's page. Cross-audience navigation is solved by (a) chapter scoping and (b) prominent "if you want X, read chapter Y" callouts on every chapter landing page.

## Design principles

1. **One topic, one home.** A concept is defined in exactly one chapter and linked from others. Fees, for example, are *defined* per product (each product has its own fee model) but the *comparative fee table* lives in `reference/fee-comparison`.
2. **Progressive depth.** Each product chapter opens with a one-paragraph plain-English overview, then goes into accounts в†’ math в†’ instructions в†’ fees в†’ code. A reader can stop at any depth.
3. **Runnable code everywhere it matters.** Every instruction-level section ends with a minimal, copy-pasteable TS / Rust / Python example. Long demos live under each product's `code-demos.mdx` so the narrative isn't drowned in code.
4. **Version-pinned.** The SDK version, program IDs, and Solana cluster are named at the top of any page that shows code. Stale examples are the #1 failure mode of DeFi docs.
5. **Separate "how it works" from "how to use it."** Algorithms live in their own chapter; user flows live in their own chapter. Product chapters link both ways.
6. **Security is a first-class chapter, not a footnote.** Auditors should not have to grep the product chapters.
7. **Audience-segmented user flows.** End-user walkthroughs are grouped by reader role вЂ” Traders, Liquidity Providers, Token Creators вЂ” not by program or by feature.

## Top-level navigation

The site is organized into eight top-level tabs (see `docs.json`). Each tab maps to a chapter or chapter cluster on disk:

| Tab | Audience | What it contains |
|-----|----------|------------------|
| **Overview** | All | `introduction/`, `protocol-overview/` вЂ” what Raydium is and how the program graph fits together. |
| **User Guide** | End users | `getting-started/`, `user-flows/` вЂ” onboarding plus task-oriented walkthroughs grouped by reader role (Traders / Liquidity Providers / Token Creators). |
| **Products** | Devs, Researchers | `products/`, `algorithms/` вЂ” one sub-chapter per program, plus the deep math chapter. |
| **Launch tools** | Creators, Devs | `products/launchlab/`, `user-flows/launch-token-launchlab`, `products/launchlab/tips-and-gotchas/` вЂ” the LaunchLab program plus its operational gotchas. |
| **Developer Guide** | Devs | `solana-fundamentals/`, `quick-start/`, `sdk-api/`, `integration-guides/` вЂ” fundamentals, hands-on quickstarts, SDK/API reference, and integration patterns. |
| **Security & Ops** | Researchers, Operators | `security/`, `ray/` вЂ” audits, risk surface, attack vectors, plus RAY tokenomics, treasury, buybacks, staking, and protocol-revenue economics. |
| **API Reference** | Devs | `api-reference/` вЂ” per-service overview pages (REST v1/v3, Trade, Perps, LaunchLab service APIs, Owner, Dynamic IPFS, Temp). |
| **Reference** | All | `reference/`, `resources/` вЂ” program addresses, error codes, glossary, fee comparison, changelog, brand kit, community tools. |

## Per-product template

To keep the product lines comparable, every product folder follows the same shape. Programs that don't expose on-chain accounts (e.g., Perps, which is a white-label backend over Orderly Network) substitute equivalent surface-specific files.

| File | Purpose | Required for |
|------|---------|--------------|
| `index.mdx` | Chapter landing: one-paragraph intro, when-to-use, links to sub-pages, key program ID. | All |
| `overview.mdx` | Conceptual model вЂ” what this product does and why, accompanied by a diagram. | On-chain products |
| `accounts.mdx` | Every on-chain account: PDA seeds, field layout, invariants, reference to IDL. | On-chain products |
| `math.mdx` | Invariants, pricing formulas, numerical examples. Cross-links into `algorithms/`. | On-chain products |
| `instructions.mdx` | Full instruction reference: args, account list, pre/post conditions, error codes. | On-chain products |
| `fees.mdx` | Fee tiers, fee splits (LP / protocol / fund / creator); for CLMM, also reward emissions, since CLMM's reward system reuses per-position fee-growth accounting. | All |
| `code-demos.mdx` | Copy-pasteable TypeScript + Rust examples for every flow exposed by the product. | All |

Per-product extensions, where the standard template isn't enough:

- **CLMM** adds `ticks-and-positions.mdx` because tick math is too large to embed in `math.mdx`.
- **LaunchLab** adds `bonding-curve.mdx`, `creator-fees.mdx`, `global-config.mdx`, `platform-config.mdx`, `platforms.mdx`, `vesting.mdx`, plus a `tips-and-gotchas/` subfolder.
- **Perps** uses `trading-basics.mdx`, `order-types.mdx`, `collateral.mdx`, `fees.mdx`, `code-demos.mdx` вЂ” no `accounts.mdx` / `math.mdx` / `instructions.mdx` since the venue is off-chain (Orderly Network) with a Solana-side wrapper.

## Cross-chapter conventions

- **Program IDs and cluster** are always defined at `reference/program-addresses.mdx`. Other pages link there rather than hardcode values.
- **Error codes** are always defined at `reference/error-codes.mdx`. Instruction pages link to the specific anchor.
- **Math** lives in `algorithms/`. Product `math.mdx` pages give the product-specific *instantiation* and link back.
- **Fee economics** are split across two homes by intent: per-product mechanics in `products/<x>/fees.mdx`, and protocol-revenue / RAY-buyback flow in `ray/`. The cross-product comparison table lives in `reference/fee-comparison`.
- **RAY tokenomics** lives in `ray/`. Pages elsewhere that mention RAY (staking flow, fee splits, governance) link there.
- **Code versioning.** Every code block has a version banner: SDK version, program ID, Solana cluster, last-verified date.
- **Diagrams.** Architecture diagrams are SVGs checked into `/images/architecture/`. The source (Mermaid or draw.io) is committed next to the SVG.
- **User flows are audience-tagged, not feature-tagged.** A walkthrough belongs under "For Traders / For Liquidity Providers / For Token Creators" вЂ” *not* under the program it touches.

## Multi-language

The default locale (English) lives at the repo root with no path prefix. Each non-English locale lives under its own top-level folder (`zh/`, `zh-Hant/`, `ja/`, `ko/`, `ru/`, `es/`, `de/`, `fr/`) and mirrors the English tree.

- **Content (MDX):** kept in sync by Mintlify's auto-translation. New / changed English pages are translated and pushed to each locale folder automatically.
- **Navigation (`docs.json`):** **not** auto-translated. Each language has its own `tabs` block under `navigation.languages`, and group labels, tab names, and page lists must be updated by hand whenever the English nav changes structurally.
- **Tab and group labels** are in the locale's own language; **page paths** are prefixed with the locale code (e.g., `zh/products/cpmm/overview`).
- **Legacy URL handling.** `/en/<path>` redirects to `/<path>` (configured in `docs.json` `redirects`). When pages move or are renamed, add a redirect entry rather than relying on the auto-translator.

## What is explicitly out of scope

The following are *not* covered in this documentation set and should be declined if the user asks for them in-docs:

- Trading strategy / market advice.
- Token-price predictions.
- Detailed tokenomics modeling beyond what is needed to read the code.
- Internal runbooks containing secrets or admin procedures вЂ” those live in a private repo.

## Status

This is v0.2 of the architecture. Compared to v0.1:

- The old `operations/` chapter (DAO, treasury, fee parameters, emergency procedures) has been replaced by `ray/`, scoped around RAY tokenomics and protocol revenues, and moved into the **Security & Ops** tab.
- Three new product lines вЂ” `perps/`, `stable/`, `routing/` вЂ” are first-class entries under `products/`.
- `getting-started/`, `quick-start/`, `resources/`, and `api-reference/` are now first-class top-level chapters.
- `user-flows/` is reorganized in the navigation by audience role (Traders / Liquidity Providers / Token Creators) rather than by feature.
- LaunchLab now has a dedicated **Launch tools** tab and a `tips-and-gotchas/` subfolder for operational edge cases.
- Multi-language support has been formalized: English is the default at the repo root, with eight locale mirrors maintained by Mintlify auto-translation for content and by hand for navigation.

Leaf-page authoring continues chapter by chapter. Stubs that exist as scaffolding are marked with a `status: draft` note in their frontmatter.
