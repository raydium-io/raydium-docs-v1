## Назначение этого документа

Эта страница служит **источником истины** для организации документации Raydium. Перед написанием детального контента для любого раздела обратитесь к этому документу — здесь определены аудиторные потоки, границы разделов, кросс-ссылки, которые должны оставаться согласованными, и расположение файлов на диске.

Это намеренно внутренний, структурный документ. Страница, предназначенная для читателей, находится в `index.mdx`.

## Модель аудитории

Документация одновременно обслуживает трёх читателей. Каждый раздел ниже помечен тегом целевой аудитории, чтобы авторы знали нужный стиль и глубину изложения.

<CardGroup cols={3}>
  <Card title="Разработчики / Интеграторы" icon="code">
    Разработчики смарт-контрактов, интеграторы DApp/агрегаторов, пользователи SDK. Нужны структуры данных на цепи, инструкции, REST/Trade API, SDK и рабочий код.
  </Card>
  <Card title="Исследователи протокола / Аудиторы" icon="flask">
    Исследователи, аудиторы безопасности, аналитики DeFi. Нужны алгоритмы, инварианты, машины состояний аккаунтов, экономика комиссий, токеномика RAY и поверхность рисков.
  </Card>
  <Card title="Конечные пользователи" icon="user">
    Поставщики ликвидности, трейдеры (спот и перпетуалы), создатели токенов. Нужны пошаговые инструкции: обмен, добавление/удаление ликвидности, создание пула, получение вознаграждений, торговля перпетуалами, запуск токена через LaunchLab.
  </Card>
</CardGroup>

Читатель одного потока никогда не должен чувствовать себя потеряным на странице другого потока. Навигация между аудиториями решается (a) областью разделов и (b) заметными вызовами вида «если вам нужно X, читайте раздел Y» на каждой посадочной странице раздела.

## Принципы дизайна

1. **Одна тема, один дом.** Понятие определяется ровно в одном разделе и связывается в других местах. Комиссии, например, *определяются* для каждого продукта (у каждого своя модель), но *сравнительная таблица комиссий* находится в `reference/fee-comparison`.
2. **Прогрессирующая глубина.** Каждый раздел продукта начинается с однопараграфного обзора на простом языке, затем переходит к аккаунтам → математике → инструкциям → комиссиям → коду. Читатель может остановиться на любом уровне.
3. **Рабочий код везде, где это имеет значение.** Каждый раздел уровня инструкции заканчивается минимальным, готовым к копированию примером на TS / Rust / Python. Длинные демонстрации находятся в `code-demos.mdx` каждого продукта, чтобы нарратив не был затоплен кодом.
4. **Привязано к версиям.** Версия SDK, ID программ и кластер Solana указаны в верхней части любой страницы с кодом. Устаревшие примеры — главный источник отказов в документации DeFi.
5. **Отделите «как это работает» от «как это использовать».** Алгоритмы живут в собственном разделе; пользовательские потоки — в своём разделе. Разделы продуктов ссылаются в обе стороны.
6. **Безопасность — раздел первого класса, не сноска.** Аудиторы не должны искать информацию в разделах о продуктах.
7. **Пользовательские потоки, разделённые по аудитории.** Пошаговые инструкции для конечных пользователей сгруппированы по роли читателя — Трейдеры, Поставщики ликвидности, Создатели токенов — а не по программе или функции.

## Навигация верхнего уровня

Сайт организован в восемь вкладок верхнего уровня (см. `docs.json`). Каждая вкладка соответствует разделу или кластеру разделов на диске:

| Вкладка | Аудитория | Содержание |
|---------|-----------|-----------|
| **Overview** | Все | `introduction/`, `protocol-overview/` — что такое Raydium и как граф программ связывается вместе. |
| **User Guide** | Конечные пользователи | `getting-started/`, `user-flows/` — вводная информация плюс пошаговые инструкции, сгруппированные по роли читателя (Трейдеры / Поставщики ликвидности / Создатели токенов). |
| **Products** | Разработчики, исследователи | `products/`, `algorithms/` — один подраздел на программу плюс раздел с глубокой математикой. |
| **Launch tools** | Создатели, разработчики | `products/launchlab/`, `user-flows/launch-token-launchlab`, `products/launchlab/tips-and-gotchas/` — программа LaunchLab плюс её операционные особенности. |

## Шаблон на продукт

Чтобы сделать линейки продуктов сопоставимыми, каждая папка продукта следует той же структуре. Программы, которые не открывают на-цепные аккаунты (например, Perps, который является белой меткой над Orderly Network), заменяют эквивалентные файлы поверхности.

| Файл | Назначение | Обязателен для |
|------|-----------|-----------------|
| `index.mdx` | Посадочная страница раздела: однопараграфное введение, когда использовать, ссылки на подстраницы, ключевой ID программы. | Все |
| `overview.mdx` | Концептуальная модель — что делает этот продукт и почему, сопровождается диаграммой. | На-цепные продукты |
| `accounts.mdx` | Каждый на-цепный аккаунт: семена PDA, раскладка полей, инварианты, ссылка на IDL. | На-цепные продукты |
| `math.mdx` | Инварианты, формулы ценообразования, числовые примеры. Кросс-ссылки в `algorithms/`. | На-цепные продукты |
| `instructions.mdx` | Полный справочник инструкций: аргументы, список аккаунтов, пред/пост-условия, коды ошибок. | На-цепные продукты |
| `fees.mdx` | Ярусы комиссий, разделение комиссий (LP / протокол / фонд / создатель); для CLMM, также вознаграждение эмиссии, так как система вознаграждений CLMM переиспользует счёт роста комиссий на позицию. | Все |
| `code-demos.mdx` | Копируемые примеры TypeScript + Rust для каждого потока, предоставляемого продуктом. | Все |

Расширения для продуктов, когда стандартный шаблон недостаточен:

- **CLMM** добавляет `ticks-and-positions.mdx`, потому что математика тиков слишком велика, чтобы встроить в `math.mdx`.
- **LaunchLab** добавляет `bonding-curve.mdx`, `creator-fees.mdx`, `global-config.mdx`, `platform-config.mdx`, `platforms.mdx`, `vesting.mdx`, плюс подпапку `tips-and-gotchas/`.
- **Perps** использует `trading-basics.mdx`, `order-types.mdx`, `collateral.mdx`, `fees.mdx`, `code-demos.mdx` — нет `accounts.mdx` / `math.mdx` / `instructions.mdx`, так как площадка находится вне цепи (Orderly Network) с оболочкой на стороне Solana.

## Соглашения между разделами

- **ID программ и кластер** всегда определяются в `reference/program-addresses.mdx`. Другие страницы ссылаются туда вместо жёсткого кодирования значений.
- **Коды ошибок** всегда определяются в `reference/error-codes.mdx`. Страницы инструкций ссылаются на конкретный якорь.
- **Математика** живёт в `algorithms/`. Страницы `math.mdx` продуктов дают конкретизацию по продукту и ссылаются обратно.
- **Экономика комиссий** разделена по двум домам по назначению: механика по продуктам в `products/<x>/fees.mdx`, экономика доходов протокола / поток выкупов RAY в `ray/`. Таблица сравнения по продуктам находится в `reference/fee-comparison`.
- **Токеномика RAY** живёт в `ray/`. Страницы в других местах, упоминающие RAY (поток стейкинга, разделение комиссий, управление), ссылаются туда.
- **Версионирование кода.** Каждый блок кода имеет баннер версии: версия SDK, ID программы, кластер Solana, дата последней проверки.
- **Диаграммы.** Архитектурные диаграммы — это SVG, внесённые в `/images/architecture/`. Источник (Mermaid или draw.io) закомичен рядом с SVG.
- **Пользовательские потоки помечены аудиторией, не функциями.** Пошаговая инструкция принадлежит разделу «For Traders / For Liquidity Providers / For Token Creators» — *не* программе, которую она затрагивает.

## Многоязычность

Локаль по умолчанию (английский язык) находится в корне репозитория без префикса пути. Каждая неанглийская локаль находится в собственной папке верхнего уровня (`zh/`, `zh-Hant/`, `ja/`, `ko/`, `ru/`, `es/`, `de/`, `fr/`) и зеркалирует английское дерево.

- **Контент (MDX):** синхронизируется auto-translation Mintlify. Новые / изменённые английские страницы автоматически переводятся и отправляются в каждую папку локали.
- **Навигация (`docs.json`):** **не** автопереводится. Каждый язык имеет свой блок `tabs` под `navigation.languages`, и метки вкладок, названия групп и списки страниц должны обновляться вручную всякий раз, когда английская навигация структурно изменяется.
- **Метки вкладок и групп** на языке локали; **пути страниц** имеют префикс кода локали (например, `ru/products/cpmm/overview`).
- **Обработка устаревших URL.** `/en/<path>` перенаправляется на `/<path>` (настраивается в `docs.json` `redirects`). Когда страницы перемещаются или переименовываются, добавьте запись перенаправления вместо полагания на auto-translator.

## Что явно находится вне области действия

Следующее *не* охватывается этим набором документации и должно быть отклонено, если пользователь просит это в документации:

- Стратегия торговли / советы по рынку.
- Предсказания цены токена.
- Подробное моделирование токеномики сверх того, что необходимо для чтения кода.
- Внутренние runbook с секретами или процедурами администратора — они находятся в приватном репозитории.

## Статус

Это v0.2 архитектуры. По сравнению с v0.1:

- Старый раздел `operations/` (DAO, казна, параметры комиссий, процедуры аварийного восстановления) заменён на `ray/`, сосредоточенный на токеномике RAY и доходах протокола, и перемещён во вкладку **Security & Ops**.
- Три новые линейки продуктов — `perps/`, `stable/`, `routing/` — являются записями первого класса в `products/`.
- `getting-started/`, `quick-start/`, `resources/` и `api-reference/` теперь разделы верхнего уровня первого класса.
- `user-flows/` переорганизована в навигации по роли аудитории (Трейдеры / Поставщики ликвидности / Создатели токенов) вместо функций.
- LaunchLab теперь имеет посвящённую вкладку **Launch tools** и подпапку `tips-and-gotchas/` для операционных граничных случаев.
- Поддержка многоязычности формализована: английский язык по умолчанию в корне репозитория, восемь локальных зеркал поддерживаются Mintlify auto-translation для контента и вручную для навигации.

Авторство листовых страниц продолжается раздел за разделом. Заглушки, которые существуют как строительные леса, отмечены примечанием `status: draft` в их frontmatter.
