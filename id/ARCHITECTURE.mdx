## Tujuan dokumen ini

Halaman ini adalah **sumber kebenaran** untuk bagaimana rangkaian dokumentasi Raydium diorganisir. Sebelum menulis konten terperinci untuk bab apa pun, konsultasikan dokumen ini вЂ” dokumen ini mendefinisikan jalur audiens, batas bab, referensi silang yang harus tetap konsisten, dan tata letak direktori di disk.

Ini adalah dokumen struktural internal yang disengaja. Halaman pendaratan yang menghadap pembaca berada di `index.mdx`.

## Model audiens

Dokumen ini melayani tiga audiens secara paralel. Setiap bab diberi tag di bawah dengan audiens mana yang terutama dilayaninya, sehingga penulis tahu nada dan kedalaman yang ditargetkan.

<CardGroup cols={3}>
  <Card title="Pengembang / Integrator" icon="code">
    Pengembang smart contract, integrator DApp/aggregator, pengguna SDK. Membutuhkan tata letak akun on-chain, instruksi, API REST/Trade, SDK, dan kode yang dapat dijalankan.
  </Card>
  <Card title="Peneliti Protokol / Auditor" icon="flask">
    Peneliti, auditor keamanan, analis DeFi. Membutuhkan algoritma, invarian, akun/mesin status, ekonomi biaya, tokenomik RAY, dan permukaan risiko.
  </Card>
  <Card title="Pengguna akhir" icon="user">
    Penyedia likuiditas, trader (spot dan perp), dan kreator token. Membutuhkan panduan berorientasi pada tugas: swap, tambah/hapus likuiditas, buat pool, klaim hadiah, trade perp, luncurkan token melalui LaunchLab.
  </Card>
</CardGroup>

Pembaca dari satu jalur tidak boleh merasa tersesat di halaman jalur lain. Navigasi lintas audiens diselesaikan oleh (a) penentuan cakupan bab dan (b) callout "jika Anda ingin X, baca bab Y" yang menonjol di setiap halaman pendaratan bab.

## Prinsip desain

1. **Satu topik, satu rumah.** Konsep didefinisikan dalam tepat satu bab dan ditautkan dari yang lain. Biaya, misalnya, *didefinisikan* per produk (setiap produk memiliki model biayanya sendiri) tetapi *tabel perbandingan biaya* berada di `reference/fee-comparison`.
2. **Kedalaman progresif.** Setiap bab produk dibuka dengan gambaran umum plain-English satu paragraf, kemudian masuk ke akun в†’ matematika в†’ instruksi в†’ biaya в†’ kode. Pembaca dapat berhenti di kedalaman apa pun.
3. **Kode yang dapat dijalankan di mana-mana penting.** Setiap bagian tingkat instruksi diakhiri dengan contoh TS / Rust / Python minimal yang dapat disalin. Demo panjang hidup di bawah `code-demos.mdx` setiap produk sehingga narasi tidak tenggelam dalam kode.
4. **Versi yang disematkan.** Versi SDK, ID program, dan cluster Solana dinamai di bagian atas halaman apa pun yang menampilkan kode. Contoh yang ketinggalan zaman adalah mode kegagalan #1 dari dokumen DeFi.
5. **Pisahkan "cara kerjanya" dari "cara menggunakannya".** Algoritma hidup di babnya sendiri; alur pengguna hidup di babnya sendiri. Bab produk menautkan dua arah.
6. **Keamanan adalah bab kelas satu, bukan catatan kaki.** Auditor tidak boleh harus grep bab produk.
7. **Alur pengguna yang tersegmentasi audiens.** Panduan pengguna akhir dikelompokkan berdasarkan peran pembaca вЂ” Trader, Penyedia Likuiditas, Kreator Token вЂ” bukan berdasarkan program atau fitur.

## Template per-produk

Untuk menjaga lini produk tetap sebanding, setiap folder produk mengikuti bentuk yang sama. Program yang tidak mengekspos akun on-chain (misalnya, Perp, yang merupakan backend white-label di atas Orderly Network) menggantikan file khusus permukaan yang setara.

| File | Tujuan | Diperlukan untuk |
|------|--------|------------------|
| `index.mdx` | Pendaratan bab: intro satu paragraf, kapan digunakan, link ke sub-halaman, ID program kunci. | Semua |
| `overview.mdx` | Model konseptual вЂ” apa yang dilakukan produk ini dan mengapa, disertai diagram. | Produk on-chain |
| `accounts.mdx` | Setiap akun on-chain: seed PDA, tata letak field, invarian, referensi ke IDL. | Produk on-chain |
| `math.mdx` | Invarian, formula harga, contoh numerik. Cross-link ke `algorithms/`. | Produk on-chain |
| `instructions.mdx` | Referensi instruksi lengkap: arg, daftar akun, kondisi pra/pasca, kode kesalahan. | Produk on-chain |
| `fees.mdx` | Tingkat biaya, pembagian biaya (LP / protokol / dana / kreator); untuk CLMM, juga emisi hadiah, karena sistem hadiah CLMM menggunakan kembali akuntansi pertumbuhan biaya per-posisi. | Semua |
| `code-demos.mdx` | Contoh TypeScript + Rust yang dapat disalin untuk setiap alur yang diekspos oleh produk. | Semua |

Ekstensi per-produk, ketika template standar tidak cukup:

- **CLMM** menambahkan `ticks-and-positions.mdx` karena matematika tick terlalu besar untuk disematkan dalam `math.mdx`.
- **LaunchLab** menambahkan `bonding-curve.mdx`, `creator-fees.mdx`, `global-config.mdx`, `platform-config.mdx`, `platforms.mdx`, `vesting.mdx`, ditambah subfolder `tips-and-gotchas/`.
- **Perp** menggunakan `trading-basics.mdx`, `order-types.mdx`, `collateral.mdx`, `fees.mdx`, `code-demos.mdx` вЂ” tidak ada `accounts.mdx` / `math.mdx` / `instructions.mdx` karena venue adalah off-chain (Orderly Network) dengan wrapper sisi Solana.

## Konvensi lintas bab

- **ID Program dan cluster** selalu didefinisikan di `reference/program-addresses.mdx`. Halaman lain menautkan di sana daripada hardcode nilai.
- **Kode kesalahan** selalu didefinisikan di `reference/error-codes.mdx`. Halaman instruksi menautkan ke jangkar spesifik.
- **Matematika** hidup di `algorithms/`. Halaman produk `math.mdx` memberikan *instansiasi* khusus produk dan menautkan kembali.
- **Ekonomi biaya** dibagi di dua rumah berdasarkan niat: mekanik per-produk di `products/<x>/fees.mdx`, dan pendapatan-protokol / alur buyback-RAY di `ray/`. Tabel perbandingan lintas produk hidup di `reference/fee-comparison`.
- **Tokenomik RAY** hidup di `ray/`. Halaman lain yang menyebutkan RAY (alur staking, pembagian biaya, tata kelola) menautkan di sana.
- **Versioning kode.** Setiap blok kode memiliki spanduk versi: versi SDK, ID program, cluster Solana, tanggal verifikasi terakhir.
- **Diagram.** Diagram arsitektur adalah SVG yang diperiksa ke `/images/architecture/`. Sumber (Mermaid atau draw.io) dilakukan bersama SVG.
- **Alur pengguna diberi tag audiens, bukan fitur.** Panduan milik "Untuk Trader / Untuk Penyedia Likuiditas / Untuk Kreator Token" вЂ” *bukan* di bawah program yang disentuhnya.

## Multi-bahasa

Lokal default (Inggris) hidup di root repo tanpa awalan path. Setiap lokal non-Inggris hidup di bawah folder tingkat atasnya sendiri (`zh/`, `zh-Hant/`, `ja/`, `ko/`, `ru/`, `es/`, `de/`, `fr/`) dan mencerminkan pohon Inggris.

- **Konten (MDX):** dijaga tetap sinkron oleh penterjemah otomatis Mintlify. Halaman Inggris baru/berubah diterjemahkan dan didorong ke setiap folder lokal secara otomatis.
- **Navigasi (`docs.json`):** **bukan** penterjemah otomatis. Setiap bahasa memiliki blok `tabs` sendiri di bawah `navigation.languages`, dan label tab, nama grup, dan daftar halaman harus diperbarui dengan tangan setiap kali nav Inggris berubah secara struktural.
- **Label tab dan grup** dalam bahasa lokal sendiri; **path halaman** diawali dengan kode lokal (misalnya, `zh/products/cpmm/overview`).
- **Penanganan URL legacy.** `/en/<path>` dialihkan ke `/<path>` (dikonfigurasi dalam `docs.json` `redirects`). Ketika halaman pindah atau ditamakanulang, tambahkan entri pengalihan daripada mengandalkan penterjemah otomatis.

Berikut ini adalah *tidak* dicakup dalam rangkaian dokumentasi ini dan harus ditolak jika pengguna memintanya dalam dokumen:
